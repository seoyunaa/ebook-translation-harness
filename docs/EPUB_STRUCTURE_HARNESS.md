# EPUB 구조 하네스

## 목적

EPUB의 nav.xhtml과 toc.ncx가 서로 일치해도 둘 다 잘못된 목차일 수 있습니다. 따라서 “파일 형식이 정상이다”와 “책의 의미 구조가 정상이다”를 따로 검사해야 합니다.

이 하네스는 다음 네 종류의 상태를 구분합니다.

1. **원본에서 관찰한 근거**: source_outline.json, 원본 북마크, 인쇄 목차
2. **사람이 검토한 기대값**: toc_contract.json
3. **도구가 생성한 산출물**: combined Markdown과 임시 EPUB
4. **검증된 출판본**: 형식 검사와 목차 계약 검사를 모두 통과한 최종 EPUB

자동 추출 결과를 곧바로 정답으로 취급하지 않는 것이 가장 중요합니다.

## 권장 제목 계층

- H1: 책 제목 한 번
- H2: 부, 독립된 서문·결론·부록·주석·참고문헌·색인
- H3: 부 아래의 장, 또는 책이 정한 두 번째 계층
- H4: 장 아래의 절
- H5 이하 또는 일반 문단: 더 작은 본문 구획, 각주 묶음, 작업 메모

모든 책이 같은 모양은 아닙니다. 부가 없는 책에서는 장이 H2가 될 수 있습니다. 핵심은 Markdown 제목 수준 자체가 아니라 **검토한 계약과 실제 독자 목차가 동일한가**입니다.

## source outline은 근거입니다

EPUB 준비 단계는 가능한 경우 nav.xhtml을 먼저 읽고, 없으면 toc.ncx를 읽어 원본 목차를 source_outline.json으로 저장합니다.

이 파일은 다음 질문에 답할 때 사용합니다.

- 원본에 부가 있는가?
- 장은 어느 부 아래에 있는가?
- 서문, 결론, 부록, 주석은 독립 항목인가?
- 원본 목차가 실제 본문 범위와 일치하는가?

source outline은 자동 추출물이므로 다음 문제가 있을 수 있습니다.

- 출판사가 일부 절을 생략했을 수 있음
- 스캔 PDF에는 의미 있는 목차 데이터가 없을 수 있음
- 인쇄 목차와 전자책 목차가 다를 수 있음
- 현재 확보된 번역 범위가 원본 전체보다 짧을 수 있음

그러므로 source outline을 확인한 뒤 인쇄 목차와 실제 본문 파일을 대조하고, 그 결과로 별도의 toc_contract.json을 작성합니다.

## 목차 계약

계약은 독자에게 보여야 할 제목, 순서, 계층을 정확히 적습니다.

~~~json
{
  "version": 1,
  "items": [
    {
      "label": "제1부: 첫 번째 여정",
      "children": [
        {
          "label": "제1장: 닫힌 문",
          "children": []
        }
      ]
    }
  ],
  "forbidden_label_patterns": [
    "\\s계속\\s+\\d+$",
    "^[A-Z]\\.?$",
    "AI\\s*(?:번역|윤문)"
  ]
}
~~~

schemas/toc_contract.schema.json은 계약의 기계 판독 형식을 설명합니다. 다음 규칙은 JSON Schema만으로 충분히 표현하기 어려우므로 빌더도 별도로 검사합니다.

- 같은 부모 아래에 같은 제목을 두 번 둘 수 없음
- 생성된 목차 전체가 계약과 정확히 같아야 함
- 금지 패턴에 맞는 독자용 제목이 하나라도 있으면 실패

## 계약 검토 체크리스트

계약을 승인하기 전에 다음을 직접 확인합니다.

1. source outline 또는 원본 인쇄 목차를 보았는가?
2. 실제로 확보된 번역 조각에 모든 계약 항목의 본문이 있는가?
3. 부·장·절의 부모·자식 관계가 맞는가?
4. 번호가 빠지거나 되풀이되지 않았는가?
5. 서문, 결론, 부록, 주석, 참고문헌, 색인의 위치가 맞는가?
6. PDF 페이지 번호, 러닝헤드, 작업 ID, “계속” 표식이 빠졌는가?
7. 각주 안에서 반복된 장 제목이 목차로 승격되지 않았는가?
8. 그림·지도·도판 제목과 색인 A–Z가 불필요하게 목차에 들어가지 않았는가?
9. 원문이 없는 부분을 새로 만들지 않았는가?

## AI 번역 고지

최종 combined Markdown에는 다음 고지를 정확히 한 번 둡니다.

    > 이 전자책은 AI 윤문 번역본입니다.

이 고지는 일반 문단 또는 인용문이어야 합니다. 제목 기호를 붙이면 목차로 들어갈 수 있으므로 H1, H2, H3, H4로 만들지 않습니다.

금지 예:

    ## AI 윤문 번역

허용 예:

    > 이 전자책은 AI 윤문 번역본입니다.

## 지정 빌드가 안전한 이유

한 권을 지정하는 빌드는 다음 순서로 작동합니다.

1. combined Markdown을 읽습니다.
2. 독자용 목차 트리를 계산합니다.
3. toc_contract.json이 존재하는지 확인합니다.
4. 실제 트리와 계약의 제목·순서·계층을 정확히 비교합니다.
5. 임시 EPUB을 만듭니다.
6. ZIP, XML, OPF, nav, NCX, 내부 링크, 메타데이터를 검사합니다.
7. EPUB 안의 실제 nav 트리를 계약과 다시 비교합니다.
8. 모든 검사가 통과한 경우에만 임시 파일을 최종 경로로 원자적으로 교체합니다.

즉, 새 빌드가 중간에 실패하면 기존의 검증된 EPUB은 그대로 남습니다.

~~~powershell
python scripts/build_epubs_from_combined.py --only sample_book_ko
~~~

--only를 생략한 전체 빌드는 일상적인 수정에 권장하지 않습니다. 실수로 관계없는 책을 다시 만들 수 있기 때문입니다.

## 최종 검증

~~~powershell
python scripts/validate_epub_toc_contract.py --book-key sample_book_ko --label sample_book_final
~~~

최종 검증은 최소한 다음을 확인해야 합니다.

- combined Markdown과 EPUB이 모두 존재함
- AI 고지가 정확히 한 번 있음
- EPUB 형식과 내부 링크가 정상임
- nav.xhtml의 실제 트리가 계약과 정확히 같음
- 금지된 목차 제목이 없음
- 제목, 저자, 언어 같은 메타데이터가 정상임

## 실패했을 때

| 증상 | 가능한 원인 | 안전한 조치 |
|---|---|---|
| 계약 파일 없음 | 아직 구조 검토를 하지 않음 | source outline과 실제 본문을 확인하고 계약 작성 |
| 실제 트리와 계약이 다름 | 제목 수준, 순서, 누락, 중복 문제 | combined 생성 규칙을 고치고 다시 검사 |
| 금지 제목 발견 | 작업 ID, 페이지 표식, “계속”, AI 제목 유입 | 해당 표식을 제목이 아닌 메타데이터나 일반 문단으로 내림 |
| 장이 계약에 있지만 본문 없음 | 원본 또는 번역 범위 부족 | SOURCE GAP으로 보고하고 계약을 현재 범위에 맞춤 |
| 임시 EPUB 검증 실패 | 링크, XML, OPF 또는 메타데이터 오류 | 최종본을 덮어쓰지 말고 원인을 수정 |

실패를 억지로 우회하거나 빈 장을 만들어 통과시키면 안 됩니다.

## SOURCE GAP 처리

원본 목차에 장이 있지만 그 장의 원문 파일이 없을 수 있습니다. 이때는 다음처럼 처리합니다.

1. 확보한 파일 목록과 원본 목차를 비교합니다.
2. 없는 범위를 SOURCE GAP으로 보고합니다.
3. 현재 확보된 범위만 정확하게 구조화합니다.
4. 사용자가 원문을 추가로 제공한 뒤 별도 작업으로 이어갑니다.

목차만 보고 본문을 추정하거나 생성하지 않습니다.

## 공개 저장소 위생

공개 저장소에는 하네스 코드, 테스트, 일반 문서, 스키마, 완전한 가상 예시만 둡니다.

다음은 공개하지 않습니다.

- 원본 EPUB, PDF, 이미지
- 원문 추출 조각
- 번역·윤문 조각
- combined Markdown과 완성 EPUB
- 실제 책의 source outline, 목차 계약, 메타데이터
- 개인 로컬 경로가 들어간 로그와 보고서

.gitignore는 1차 방어선일 뿐입니다. 커밋 직전에는 반드시 git status와 git diff --cached를 확인해 실제로 올라가는 파일을 사람이 검토해야 합니다.
